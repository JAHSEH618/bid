# 使用指南

> 面向实际写标书的用户。从第一次登录到导出 .docx，按操作顺序写。

## 0. 环境准备

确认运维已经部署完成（见 [`app/README.md`](app/README.md)），你拿到了：

- 系统访问地址：例如 `http://10.x.x.x:12123` 或域名
- 默认管理员账号：`admin / admin123`
- 一份阿里云百炼（DashScope）API Key（形如 `sk-xxxxxxxx`）

> **百炼 API Key 哪来的**？登录 <https://bailian.console.aliyun.com/> → 我的应用 → API-KEY 管理 → 创建。建议给本平台单独开一把 Key 方便统计 token 消费。

---

## 1. 第一次登录与改密

1. 浏览器打开系统地址，登录页输入 `admin / admin123`
2. 系统会强制跳转到**修改密码**页（首次登录必改）
3. 新密码要求：**至少 8 位 + 含字母 + 含数字**
4. 改密成功 → **自动跳回登录页** → 用新密码登录

> ⚠️ 默认 admin 是初始管理员。生产环境建议改密后立即在 **管理后台**新建普通用户分给团队成员，不要让所有人共用 admin。

---

## 2. 配置百炼 API Key（必须做一次）

1. 顶部导航栏 → **设置**
2. 在 "API Key 配置" 卡片填入百炼 Key（明文输入，保存后系统用 AES-GCM 加密落库，再次显示是 `sk-***xxxx`）
3. 点 **测试连通性**
   - ✅ 显示 "API Key 有效" → 可以用
   - ❌ 显示 "认证失败" → Key 写错或被禁用，回百炼控制台核对
4. 同页可以看 **本月 token 消费图**（按模型分组：qwen3.6-max / qwen3.6-flash 等）

> 💡 **API Key 真快照机制**：你点"启动生成"时，系统会把当前 Key 加密快照拷到项目里。后续即使你删除 / 修改 Key，正在跑的项目仍能跑完。这是为了避免运行中失败。

---

## 3. 创建一个新项目

### 3.1 新建项目

1. 顶部导航 → **项目** → 右上角 **新建项目**
2. 填项目名（例如 "天翼支付智慧客服外包"）+ 简短描述
3. 点 **创建** → 跳转到上传页

### 3.2 上传文档（最关键的一步）

系统支持三种文档槽位，**至少传 1 份**，建议三份都传：

| 槽位 | 推荐文档 | 作用 |
|---|---|---|
| **技术规范书** | 招标方原文（technical spec） | 让 LLM 读甲方的硬性要求，作为提纲与正文的核心依据 |
| **打分规则** | 评分细则 / 资格审查标准 | LLM-1 在拟提纲时把章节与打分项一一对应，避免漏分 |
| **模板范例** | 历史中过的方案 / 行业范文 | 给 LLM 风格参考，统一术语和结构 |

支持格式：`.docx` `.doc` `.md` `.txt`（单文件 ≤ 50MB；同用户日累计 ≤ 500MB）

> ⚠️ 不支持 `.pdf`、`.ppt`、扫描件、加密 .doc。如需，先用 Word 另存为 `.docx`。系统会用 LibreOffice 把 `.doc` 转成 `.docx` 再读，第一次传 `.doc` 可能稍慢。

### 3.3 启动生成

1. 至少传完技术规范书 → 点 **启动生成**
2. 系统进入 LangGraph 工作流：
   - 抽取文档（markitdown 走 LibreOffice 转换）
   - LLM-1 拟提纲（约 1-3 分钟，看文档大小）
3. 提纲生成完 → **自动停下**，等你审核

---

## 4. 提纲确认

提纲页显示 LLM-1 拟好的章节列表（标题 / 摘要 / 关键要点 / 目标页数）：

- ✅ **直接确认沿用**：点 **确认并开始生成章节**，工作流继续
- ✏️ **编辑提纲**：可以改章节标题、摘要、要点；可以增加 / 删除章节；改完点 **确认并开始生成章节**

> 💡 提纲是后续 N 章的骨架，**这一步多花 5 分钟比后面每章 revise 划算**。重点检查：章节是否覆盖打分项？标题语义是否专业？关键要点有没有缺重要的承诺？

---

## 5. 章节审核（核心环节）

### 5.1 流式生成观察

每章生成时，**右侧主区域逐字打字**（SSE 流式）。左侧侧栏显示：

- 章节列表 + 状态徽章（pending / generating / awaiting_review / approved / skipped / failed）
- 进度条（已通过 / 总数）+ 跳过 / 失败计数

> 🔄 **刷新页面也不丢内容**：系统每秒把累积的 token 写 DB，刷新或切到其他页面再回来，看到的内容连续累积，不会从头开始。

### 5.2 等章节进入 `awaiting_review` 状态

LLM 写完一章（约 1-3 分钟）→ 进度条到该章变绿 + 出现 **三按钮**：

| 按钮 | 含义 | 影响 |
|---|---|---|
| ✅ **通过** | 这章满意，标 approved，进下一章 | retry_count 不重置 |
| ✏️ **修订** | 让 LLM 重写 | 弹出反馈框输入修改意见，retry_count + 1 |
| ⏭️ **跳过** | 这章不要了，标 skipped，进下一章 | 后续 docx 不含这章 |

### 5.3 修订（revise）的关键

点 **修订** 后弹出反馈框，**写得越具体效果越好**：

- ✅ 好：「第 3 段服务流程要分 5 步：受理 → 派单 → 处置 → 反馈 → 闭环；每步给出 SLA 时效」
- ✅ 好：「在表格里加一列 ‘对应打分项’，对齐评分细则 §2.3」
- ✅ 好：「整章太空泛，针对甲方业务规模 300 坐席 + 7×24 小时给出具体配比」
- ❌ 不好：「不行，重写」（LLM 不知道哪不行）
- ❌ 不好：「写好一点」（同上）

> 💡 **revise 是 patch 模式**：LLM 同时拿到 **上一轮正文 + 你的修改意见**。局部建议（"加表格" / "改流程图"）会保留其他部分；整体建议（"内容空泛" / "逻辑混乱"）会做实质重组。**不会**简单换措辞。

修订有最大次数（默认每章 3 次）。第 4 次自动跳到 skipped，避免死循环。

### 5.4 章节失败 → retry

如果 LLM 调用超时 / 网络错 / 内容审核拦截，章节会标 `failed` 红色徽章 + **重试按钮**。点重试：

- retry_count 归 0（避免被前几次错的次数算进去）
- 之前那一轮的版本标 abandoned（保留在历史 Tab，可见但不参与最终方案）
- 重新启动 LLM 生成

> 💡 **历史版本 Tab**：点章节标题旁的 "📜 历史" 可以看这章被 revise / retry 过多少次的所有版本。

## 6. 全文预览与导出

### 6.1 全文预览

侧栏所有章节 approved 或 skipped → 系统自动跳到 **全文预览**：

- 完整 Markdown 渲染（含 Mermaid 图、表格、代码块）
- Mermaid 图客户端实时渲染（白底，中文字体已配）

### 6.2 导出 .docx

页面底部 **导出面板**：

| 操作 | 说明 |
|---|---|
| **复制全文** | 复制 Markdown 源码到剪贴板（粘贴到任意 Markdown 编辑器） |
| **下载 .md** | 下载完整 Markdown 文件（GitHub Flavored 兼容） |
| **下载 .docx** | 触发后端生成 Word，进度条显示 4 阶段：rendering_mermaid → pandoc → finalizing → done |

`.docx` 文件名格式：**`项目名_技术方案_YYYYMMDD.docx`**（中文 UTF-8 + ASCII fallback）

### 6.3 二次美化（推荐）

`.docx` 是 Pandoc 自动排版，**给排版同事二次美化 30 分钟**能出片：

- 统一字体（仿宋 GB2312 / 黑体）
- 调整段落间距、行距 1.5
- 表格边框、居中
- 添加封面 / 目录 / 页眉页脚
- 公司 logo 水印

## 7. 多人协作（团队共享池）

### 7.1 项目可见性

**所有用户**默认可见**所有项目**（共享池）。但操作权限分级：

| 操作 | 谁能做 |
|---|---|
| 查看项目 | 所有登录用户 |
| 上传 / 启动 / 修订 / 重试 | 项目创建者 + 任何用户（共享） |
| 删除项目 | 项目创建者 + admin |
| 用户增删 / 重置密码 | admin |

### 7.2 协作建议

- **谁创建谁负责**：尽量让一个人负责一个项目从头到尾，避免审核冲突
- **多窗口隔离**：同事可以在另一项目并行开工（系统支持并发 ≤ 10 个项目）
- **修订意见集中**：写标书一般有 1 个总负责人，TA 来点三按钮；其他人可以在飞书 / 微信里收集反馈给 TA 集中输入
- **TIP**：多人同时审一个章节会有冲突（最后点的覆盖前面的），建议团队约定"哪个项目谁审"

---

## 8. 管理后台（admin 专用）

顶部导航 → **管理**（只对 role=admin 可见）：

### 8.1 用户管理

- 新建用户（用户名 + 临时密码 + 角色 admin/user）→ 用户首次登录强制改密
- 启用 / 禁用用户（禁用后无法登录但保留历史）
- 重置密码（系统不显示原密码，admin 设新临时密码并通知用户）
- 删除用户（如果 TA 创建过项目，会被 RESTRICT 阻止；先把项目转给别人 / 删项目再删人）

### 8.2 token 消费汇总

按月看全员 token 消费：

- 按模型分组（qwen3.6-max / qwen3.6-flash 等单价不同）
- 按用户分组（看谁烧得多）
- 用于费用分摊或预算控制

## 9. 常见问题

### Q1：章节生成卡了 5 分钟没反应

**第一步**：刷新页面看状态。

- 如果显示 `failed` 红色徽章 + retry 按钮 → 点 retry，系统会自动从 LangGraph checkpoint 续跑
- 如果还是 `generating` → 后端 cron 每 3 分钟扫一次 stale 章节，超过阈值会自动标 failed
- 如果仍卡 → 联系运维看 docker logs

### Q2：导出 .docx 进度卡 finalizing

DOCX 状态机有四处自动 repair：cron / POST cached / GET docx-job / GET 下载。任意刷新页面或重新点下载都会触发 repair。如果还卡，看后端 logs。

### Q3：Mermaid 图显示"图表渲染失败"

LLM 偶尔生成的语法新版 mermaid 不认。失败卡片下方有 **mermaid.live 一键调试链接**，点开可以手动调整。或者在 **修订** 时告诉 LLM"把 X 那张图改成简单的 flowchart TD"。

### Q4：重新生成后内容重复 / 啰嗦

revise 反馈写得不够具体。换一种写法：
- ❌ "内容多一点"
- ✅ "在 3.2 节加一段管理人员激励机制，含晋升通道、培训预算、团建活动费用 100 元/人/月"

### Q5：忘了密码怎么办

找 admin 用 **管理后台 → 重置密码** 给你设新的临时密码，再次登录强制改密。如果连 admin 都忘了，运维可以直接 SQL UPDATE（见 `app/README.md` 故障排查）。

### Q6：能不能跑同一项目的两个并行版本（A/B 比较）

**不能**。每个项目只有一个 LangGraph thread。如果想 A/B：复制项目（同样的文档传第二次），用不同 revise 策略各跑一次。

### Q7：百炼 API Key 余额不足 / 限流

LLM 调用会自动重试 2 次（每次间隔 2-5s）。如果还失败，章节标 failed → 你充值后点 retry。

### Q8：上传文档后启动失败"markitdown 不支持该格式"

通常是：扫描 PDF / 加密 .doc / 损坏文件 / 格式错乱。系统不会 silent 跳过（会显式 fail），自己用 Word 打开能看到内容才说明文件 OK。

## 10. 数据与备份

- 你创建的项目数据存在服务器 `/var/lib/bid-app/projects/{project_id}/`
- 每天凌晨 3 点 cron 会做 `pg_dump` 全量备份到 `/var/lib/bid-app/backups/`，滚动保留 7 天
- 删除项目会**级联删除**磁盘文件 + DB 行 + DocxJob + ChapterVersion + token_usage（不可恢复）
- 想保留历史方案：导出 .md + .docx 到本地存档，再删项目腾名额

---

## 附录：键盘快捷键 / 触摸板

- 登录后：`Tab` 键可以跳过顶部导航直达主内容（skip-link）
- 表单提交错误：自动 focus 到第一个错误字段
- 删除 / 重置等破坏性操作：默认聚焦"取消"按钮（防误点）
- 三按钮章节审核：建议鼠标点（按钮间距留足，不容易误触）

- 使用问题、bug 报告、新功能建议 → 提 GitHub Issue 或在团队群同步
- 后端 / 前端代码改动 → 见 [`app/IMPLEMENTATION_SPEC.md`](app/IMPLEMENTATION_SPEC.md) 的实施蓝图（~7100 行 + D-A 至 D-EE 决策表）
