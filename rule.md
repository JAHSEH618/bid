# 投标技术应答文件模版规则

> 基于《"长沙城市消费服务平台"票务营销一体化项目应答书 V1.3》提取。该文件被认定为"十分规范的模版"，本规则用于约束后续同类应答文件的生成与审校。

## 0. 文档定位

- 类型：政企信息化项目的**技术应答文件 / 应答书**，对应"竞争性磋商 / 评审"采购模式。
- 体量参考：正文段落 + 表格 ≈ 685 块；正文段落均长 62 字（中位 45 / p90 137），即**短段为主，重在结构密度**而非长篇论述。
- 视觉元素：表格 5 张（全部在前/后置评审表与人员表），图片 101 张（架构图 1、模块时序图 ≈35、案例与证书扫描件 ≈60）。

---

## 1. 文档骨架（H1 顺序固定，缺一不可）

按以下顺序输出，**顺序不可调整**：

| # | H1 标题 | 性质 | 必含元素 |
|---|---------|------|---------|
| 1 | 评审索引表 | 表格章 | 仅 1 张表，列：`序号 / 评审因素 / 应答值 / 应答文件对应页码` |
| 2 | 技术商务符合性评审细项 | 表格章 | 仅 1 张表，列：`序号 / 评审细项 / 细项应答` |
| 3 | 项目建设及技术事项方案 | **核心技术章**（占正文 ≥ 60%） | 见 §3 |
| 4 | 服务保障方案 | 管理与服务章 | 项目管理 + 售后服务两大块 |
| 5 | 系统安全保障措施 | 安全章 | 重点场景安全 + 系统维护 |
| 6 | 类似业绩 | 业绩章 | 3 个案例 + 同类规模小结，正文 ≈ 全部为合同扫描件指引 |
| 7 | 企业综合实力 | 资质章 | 4 项体系证书（信安/ITSS/质量/CMMI5），每项 2 张证书图 |
| 8 | 人员要求 | 人员章 | 项目经理表 + 核心团队表 + 核心成员证书 |
| 9 | 技术/商务响应与偏离表 | 偏离表 | 列：`序号/磋商文件条目号/采购规格/响应规格/响应与偏离/说明` |

**封面（无 H1 标号）**：4 行居中，依次为
```
"<项目名称>"<项目子标题，如：技术支撑服务采购>
应答文件
<供应商法定名称>
<YYYY年MM月>
```

## 2. 标题层级与编号

- 使用 Word 内建样式 `heading 1` … `heading 9`，**最深可达 H5**（实际 H1 9 个、H2 18 个、H3 30 个、H4 40 个、H5 7 个）。
- 标题文字**不带数字编号**（编号靠 Word 多级列表自动生成，正文里只写中文标题）。例外：业绩章 H2 内嵌业绩金额 / 时间，如 `案例1：2023客户运营应用优化提升服务合同，合同金额1502.4万元`。
- 同一父标题下的子标题数量保持**3–8 个**为宜；H4 模块下若有展开，统一展开到 H5 一层（不再下钻）。

---

## 3. §3"项目建设及技术事项方案" 内部结构（核心模板）

固定三段式 H2：

```
3.1 项目概述
    3.1.1 项目背景      ← 2–3 段叙述，回应"为什么要建"
    3.1.2 项目目标      ← 编号清单 6 条左右，每条覆盖一个核心能力
3.2 系统整体架构
    3.2.1 总体设计原则
        3.2.1.1 应用架构方案  ← 唯一一张总体架构图
    3.2.2 应用解决方案  ← 模块化展开，见 §3.1 子规则
3.3 技术实施方案
    3.3.1 核心技术栈选型
    3.3.2 系统部署方案
    3.3.3 数据迁移与业务切换方案
    3.3.4 应用数据安全方案
    3.3.5 监控与运维方案
```

### 3.1 设计原则的固定 5 条

`总体设计原则` 节正文**必须给出且只给出 5 条**，名称、顺序固定：

1. **开放性**——技术栈、接口规范、跨端能力
2. **灵活性与扩展性**——领域拆分、能力中心化
3. **稳定性**——分布式、读写分离、缓存削峰、熔断限流、灰度
4. **易维护性**——配置化、统一日志/监控/审计
5. **安全性**——分级保护、风控、合规

每条采用 `<编号>、<原则名>：<换行>` + 一整段 100–180 字描述。

### 3.2 总体架构表述

`应用架构方案` 节**必须明确写出七层架构**，命名固定为：

> 接入层、网关层、业务服务层、能力中心层、集成接口层、数据服务层、基础设施层

随后用 7 个短句逐层说明，最后**插入 1 张总体架构图**。

### 3.3 模块章（H4）的固定三段格式 ⭐

`3.2.2 应用解决方案` 下的每一个业务模块（H4）必须按以下三段结构写：

```
[H4] <模块名>
技术实现：
    <2–6 段长文，描述能力构成、关键技术选型、数据组织方式>
关键适配：
    <1–3 段，明确本项目的差异化点 / 边界 / 安全控制等>
典型业务流程：
    <每个流程一段，结构如下>
    <流程名>。流程目标是…。处理步骤为：…。关键控制点包括…。
    对应时序图：<流程名>
    [图片：时序图]
```

- 一个模块通常含 **4–8 个流程**，每个流程**配一张时序图**；图片紧跟在 `对应时序图：xxx` 行之后。
- 流程描述必含三要素，顺序固定：**`流程目标` → `处理步骤` → `关键控制点`**（可由结构化校验脚本检测，参见模版统计：39 个流程，39 处全要素命中）。
- 模块字数参考：**1400–2800 字**；体量异常大的"专题章"（如"高并发解决方案"）可不遵循三段式，改为分章节论述（参考值：≈10000 字 + 2–3 张图）。

参考模块清单（票务消费场景，可裁剪）：

1. 用户认证与权限管理模块
2. 商户活动与门店运营模块
3. 票务商品与交易模块
4. 权益、消费券与核销模块
5. 数据统计、对账与报表模块
6. 第三方平台与监管对接模块
7. 高并发解决方案 / 性能保障专题（不走三段式）

## 4. §4"服务保障方案" 子结构

```
项目管理方案
    项目管理
        项目开发与过程管理
        项目立项 → 需求分析 → 项目计划 → 系统设计 → 开发编码 → 系统测试 → 系统验收
    项目沟通管理 → 例会 / 不定期会议 / 项目状态日周报制度 / 沟通手段
    项目风险管理 → 风险来源 / 技术风险 / 组织和管理风险 / 外部风险 / 风险策略 / 风险评估
    系统验收与移交 → 验收流程 / 交付文档清单
售后服务方案
    日常支撑维护服务 → 客诉及咨询日常支撑 / 文档及培训支撑 / 疑难技术支持
    应用保障措施 → 7*24小时技术支持 / 响应时间 / 远程技术支持
```

会议描述统一采用四要素：`会议目标 / 日期与时间 / 参加人员 / 主要议程及责任`。

## 5. §5"系统安全保障措施" 子结构

```
重点场景系统安全与保障措施
    营销场景系统安全保障
    票务场景系统安全保障
    卡券场景系统安全保障
系统维护服务
    系统维护服务
    系统故障处理
```

按"业务场景"切分而非按"安全维度"切分——这是本模版的特点。

## 6. §6"类似业绩" 模板

- 至少给出 **3 个案例**，每个案例 H2 标题格式：
  ```
  案例N：<年份> <项目名>，合同金额<金额>
  ```
- 每个案例固定 4 个 H3 子项（全部为扫描件配图）：
  1. 合同服务内容页
  2. 合同附件1-技术规范书（具体服务内容页-关键字：…）
  3. 合同金额页（合同含税金额：￥xxx）
  4. 合同签订时间页、合同签页（签约时间：YYYY.MM.DD）
- 案例后追加 1 个 H2：`同类业绩系统规模`，正文用单张 H3 子项概括（如"用户规模、系统规模、系统承载量超千万级"）。
- **正文几乎不写文字**，靠扫描图说话；H3 标题本身就是说明。

## 7. §7"企业综合实力" 模板

固定 4 项资质，**顺序不变**：

1. 信息安全管理体系认证证书（ISO 27001）
2. 信息技术服务管理体系认证证书（ISO 20000 / ITSS）
3. 质量管理体系认证证书（ISO 9001）
4. CMMI5 证书

每项 2 张证书扫描件（正本 + 副本 / 中英文）。

## 9. §9"技术/商务响应与偏离表"

- 单一表格：`序号 / 磋商文件条目号 / 采购规格或商务条款 / 响应文件的规格或商务条款 / 响应与偏离 / 说明`
- 表下必附 3 条说明（原文照抄）：
  > 1. "响应与偏离"应注明"响应"或"偏离"。
  > 2. 属磋商文件规定可能变动的内容在"说明"栏中注明。
  > 3. 若全部条款无偏离，供应商需承诺"全部响应磋商文件商务和技术条款及合同条款的要求，无偏离"。
- 表后落款两行：
  ```
  供应商名称：<法定名称>
  日      期：    YYYY   年   MM   月   DD   日
  ```

## 10. 文体与措辞规范

- **语种**：简体中文。术语首次出现的英文/缩写以括号形式给出（如 `RBAC`、`JWT`、`Spring Cloud`、`API`、`PDA`），后续直接用缩写。
- **句式**：陈述句为主，避免口语化与第一人称。常见动词：`承载 / 支撑 / 沉淀 / 协同 / 统一 / 装载 / 回写 / 回流 / 收口 / 治理 / 拦截`。
- **段长**：单段不超过 **200 字**为宜（模版 p90 = 137 字）；超长说明需折成多段或用编号列表。
- **数字与单位**：金额写"￥xxx.xx 元 / xx 万元"两种并列；日期用"YYYY.MM.DD"或"YYYY 年 MM 月 DD 日"。
- **强调**：仅在极少数核心理念处使用 `**xxx**` 加粗（模版中仅出现于"一码游"等核心体验词）。
- **引号**：中文双引号 `"…"` ；模块化能力名称多用引号包裹（"统一用户体系""一码核销"）。
- **避免**：感叹号、问号、营销式口号、未量化的形容词（"非常优秀""极大提升"），所有性能/规模描述必须可被需求或案例佐证。

---

## 11. 列表与符号

- **数字列表**：使用半角阿拉伯数字 + `.` 或全角 `、`，即 `1.xxx` / `1、xxx`，**不混用一二三、不使用 ①②、不使用 ◆▶●**。模版统计：`1.` / `1、` 共 ≈ 29 处，其他符号 0。
- **二级编号**：用 `（1）（2）…`。
- **缩进**：列表项之间不空行；列表与正文段之间空一段或直接接段。

---

## 12. 表格使用规则

- 全文表格**只在 4 处**出现：评审索引表、评审细项、人员表（PM + 核心团队）、偏离表。
- **正文章节内禁止使用表格**展开内容；技术要点改用编号列表或段落小标题。
- 表头列名按 §1 / §8 / §9 给出的固定列名书写，**不要随意改名或加列**。

---

## 13. 图片使用规则

- **架构图**：仅 1 张，置于 `应用架构方案` 末尾，整体说明完之后再插图。
- **时序图**：每个 H4 模块在"典型业务流程"段为每个流程**各配 1 张**；插图前**必须**有一行 `对应时序图：<流程名>`。
- **业绩 / 资质扫描件**：紧跟其 H3 标题之后，无需文字描述；标题本身即为图注。
- 图片宽度建议保持 **页面 60% 左右居中**（与项目当前 DOCX 导出策略一致）。
- 流程图、时序图源文件如以 Mermaid 维护，最终以 PNG 形式嵌入；嵌入前**校验是否能成功渲染**，渲染失败的图必须替换或回退为源码视图，不允许留空。

## 14. 与生成器（bid-app）对齐的工程约束

> 这一节是为 LangGraph 工作流的提示工程与节点行为而设。

- **LLM-1（生成大纲）**：输出严格遵循 §1 的 9 个 H1 / §3-§9 的固定子节，禁止生成"创新点""核心亮点"等非模版章节。
- **LLM-2（章节正文）**：在 H4 模块章中**必须**输出三段式标记词 `技术实现：` / `关键适配：` / `典型业务流程：`；每个流程必含三要素 `流程目标 / 处理步骤 / 关键控制点`，并以 `对应时序图：<流程名>` 结束以触发 LLM-3。
- **LLM-3（生成可视化）**：每出现一次 `对应时序图：<流程名>` 触发一次 Mermaid `sequenceDiagram` 生成；其他场景下不主动产图；总体架构图独立生成一次（`graph TD` 七层）。
- **校验器（merge_chapter 之后）**：
  - 设计原则 = 5 条且名称匹配白名单；
  - 七层架构关键词全部命中；
  - 每个模块章三段式齐全；
  - 每个流程的三要素齐全且配一张图；
  - 业绩章每个案例的 4 项扫描件 H3 齐全；
  - 资质章 4 项体系齐全且每项 ≥ 1 张图；
  - 偏离表存在且尾部三条说明 + 落款齐全。
  任一项缺失 → 提示用户人工 revise，不静默通过。

---

## 15. 一句话总结

> **结构高于辞藻、节奏高于篇幅、可核查高于可读性。** 模版的"规范感"来自固定章节顺序、固定模块三段式、固定流程三要素和固定扫描件清单——任何输出都应能被一段确定性脚本逐项核对通过。
